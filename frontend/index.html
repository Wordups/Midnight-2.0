Architecture

Three clean layers. No bleed-through.

Extraction, mapping, and rendering are strictly separated. Each layer does one job.

01
Extraction
Reads the source document. Pulls all fields, sections, bullets, tables, and revision history. Outputs POLICY_DATA.
02
Mapping
Normalizes into a fixed schema. Maps to compliance frameworks. Identifies gaps. Generates suggested policy language.
03
Rendering
Rebuilds into the client-branded template. Handles layout only. Deterministic output regardless of content length.

Midnight's approach to policy operations is grounded in real enterprise security experience — the same methodology that defines how high-performing compliance teams actually work.

The problem with policy documentation

Most organizations manage policy documentation the same way they did in 2005. Documents live in shared drives. Frameworks are mapped manually, if at all. Auditor questions get answered by whoever happens to know which folder contains what. Gaps aren't discovered until an audit reveals them.

The problem isn't effort. It's architecture. Policy operations needs the same discipline applied to any other operational system — defined inputs, controlled processing, reliable outputs, and a searchable record of everything.

That's what Midnight is. Not a document formatter. A policy operations engine.

No guessing which policy covers what
Every policy is indexed, framework-mapped, and searchable. Auditor asks a question — you have the answer in seconds.
No gaps discovered in the audit room
Framework mapping runs on every document. Gaps surface before the auditor does, with suggested language to close them.
No manual reconstruction
Legacy documents go in. Structured, formatted, compliant output comes out. The work that used to take weeks takes minutes.
